# Application Patterns

Read the relevant pattern after the target and seed are known.

## New-project Start

Read [start-mode.md](start-mode.md). Convert the plain-language product idea into the complete `.selective-intelligence/` Start Pack before project code. Map the full product and end-state architecture, define the smallest complete MVP, and lock the current release and first vertical build slice. When building is authorized, execute only the locked slice, control material changes through amendments, and close it only after the as-built system, evidence, coverage map, and next baseline agree.

Select the micro, standard, or high-assurance profile by reversibility, blast radius, sensitive data, external effects, and operational risk—not by how impressive the idea sounds. Run the dependency-free validator before using a lock verdict.

## Continue or resume

Read [continuity-and-impact.md](continuity-and-impact.md). Recover the canonical lock, exact source revision, active build, claimed owners, partial filesystem and external effects, invalidated evidence, and next safe action. Do not repeat external actions or merge overlapping owners from memory. When the baseline changed, revalidate and re-lock before ordinary implementation resumes.

## Repository crawl and realignment

Treat a plain-language product complaint or desired outcome as the seed. Read [repository-intelligence.md](repository-intelligence.md), recover the intended user flow, build the product-to-code map, classify each feature's proven state, and trace gaps across routing, navigation, UI, backend, data, permissions, tests, configuration, and deployment. Implement the coherent correction when authorized. Remove obsolete paths that caused drift, then re-search for resurfacing risks and verify the real user journey.

## Person or performer onboarding

Example seed: an official website and the instruction to create a Sway profile.

1. Resolve the person and target account.
2. Inspect the destination profile schema and existing profile conventions.
3. Extract confirmed name, stage identity, profession, biography facts, official links, booking/contact route, location wording, and appropriate public media.
4. Follow first-party links only as needed to fill high-value profile components.
5. Write a concise destination-native biography instead of copying the source.
6. Create hierarchy, sections, calls to action, and visual treatment using product conventions.
7. Do not invent audience size, achievements, availability, rates, affiliations, or account ownership.
8. Create the profile when authorized; do not accept terms or create credentials on the subject's behalf.
9. Verify identity, links, images, mobile layout, and primary action.

The finished profile should look intentionally built, not imported from a sparse form.

## Business onboarding

Use a website, directory entry, social presence, menu, catalog, or supplied documents as seeds. Reconcile official name, location, service area, offerings, hours, contacts, policies, and media. Map them into the destination's canonical records and workflows. Avoid copying stale directory claims or creating duplicate business records. Connect the correct inquiry, booking, ordering, inventory, or discovery actions.

## Artifact reconstruction or migration

Inspect both the source and target. Preserve intent, verified content, high-performing structure, and unique assets. Reject known defects, obsolete fields, outdated copy, duplicated flows, inaccessible patterns, and source-specific workarounds. Rebuild using the target's native architecture. Validate that the old path cannot silently resurface or conflict unless compatibility is intentionally required.

## Project or system onboarding

Use repositories, briefs, issue history, documents, and current runtime state to construct the operating picture. Identify product purpose, authority, canonical architecture, active work, blockers, and verification methods. Distinguish documentation claims from observed state. Produce or update the actual project artifacts requested; do not substitute a generic audit for implementation.

## Campaign or content completion

Use the sparse brief to determine audience, offer, action, channel, brand rules, and proof. Research only missing facts that affect accuracy or persuasion. Build the complete channel-appropriate asset. Treat positioning, hierarchy, and original prose as created decisions; treat prices, guarantees, inventory, results, and affiliations as evidence-requiring facts.

## Diagnosis and recovery

Use symptoms as seeds, then inspect the smallest set of logs, code paths, configuration, recent changes, and live behavior needed to isolate the failure. Inherit only verified current state. Separate root cause from correlated noise. If asked to fix, implement and validate the real recovery; if asked only to diagnose, stop after evidence-backed findings.

## Feedback and learning

Read [feedback-and-learning-loop.md](feedback-and-learning-loop.md) after a meaningful completion, correction, retry, reopened requirement, blocked gate, or false-positive gate. Infer the outcome from observable evidence when possible. Record a tiny structured local event, not raw conversation or hidden reasoning, then promote recurring or severe causes into a reproducible eval and the narrowest effective gate.

## Self-application

Treat the user's current intent as governing above the skill's existing doctrine. Reconstruct the current skill, compare it with independent mechanisms, select one bounded coherent release, preserve protected behavior, implement the change, and forward-test from fresh context. A self-authored acceptance criterion is not independent proof.

## Cross-mode quality gate

- The target performs its intended user job.
- Sparse input did not produce sparse presentation.
- Every material fact is supported or bounded.
- Creative decisions are coherent with the destination.
- No stale or defective source behavior was inherited by accident.
- Unknown optional details do not block a useful result.
- The handoff identifies only meaningful remaining gaps.
- Meaningful outcome signals are recorded locally without assuming silence means success.

## JumpStart and Guided Council

Begin from the user's ordinary-language outcome or ask the single JumpStart question when none exists. Route continuing work to its dedicated Project when available, recover and lock intent, select existing capabilities, then execute the smallest complete Worker task. Automatically use distinct spawned agents when the environment supports them; otherwise emit the same bounded packets for fresh sequential contexts.

Challenge exact claims through a read-only Objector, disposition every finding through a distinct Aligner, correct sustained material issues, rerun invalidated proof, and stop consequential actions at the human/quorum permission and budget gate. Finish with the verified user outcome or one truthful blocker plus a Resume Packet. Suggest Project-source promotion only for permitted, approved, durable decisions or hard-won corrections.
